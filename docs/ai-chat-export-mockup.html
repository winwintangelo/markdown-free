Four UI concepts for the new AI Chat Export feature. The user pastes a public share link from ChatGPT, Claude, Gemini, DeepSeek (and other chatbots), the server fetches the conversation, and it flows into the existing PDF / DOCX / Markdown / HTML export pipeline. The final direction is recorded below; each concept is then annotated Build / Skip / Adapt.

Engineering note (does not affect this UI choice): server-side fetching of ChatGPT & Claude share pages runs into bot protection — some links will fail to fetch. Every concept below therefore includes a clear error state. The UI decision and the fetch-reliability work are independent; pick the layout you like best.
★

Final direction

Build Concept 1 + Concept 2 together, adapt one piece of Concept 4, and skip Concept 3. Concepts 1 and 2 aren't competing options — their cons are the same problem flipped, so they complete each other.

① Dedicated landing page — /ai-chat-to-pdf

The reason is positioning, not just SEO. A dedicated route makes this a first-class feature with its own value proposition — “Turn any AI chat into a polished PDF” is a sharp, shareable message. “We added a third input tab” is not. SEO coverage is a bonus on top.

② Homepage third tab — companion, not alternative

Concept 1's con (homepage users can't discover it) and Concept 2's con (still needs SEO pages) are the same problem flipped — they cancel out. The URL-input logic is already built for ①; reusing it as a tab is just a render decision. Small marginal cost, second surface.

③ Borrow Concept 4's preview — as an optional inline step

Not a forced wizard — that breaks the zero-friction flow behind the 86% upload conversion rate. But the review step matters more here than for file upload: the server-side fetch is opaque, so the user can't see what they got. So — an optional inline “► Show preview before exporting” disclosure. Confirmation value, no speed cost.

✗ Skip Concept 3 — smart auto-detect input

Design-elegant but operationally wrong. A feature invisible until you try it generates no “oh, this exists” moment. PT-007 users are actively searching for this workflow — the UI has to announce it, not hide it behind detection.
